    **Create `src/shopify_forecast_mcp/core/analytics.py`:**

    Import from `metrics.py`: `AnalyticsResult`, `AnalyticsSection`, `aggregate_metrics`, `compute_discount_rate`, `compute_units_per_order`, `SUPPORTED_METRICS`.
    Import from `timeseries.py`: `orders_to_daily_series`.

    **1. `analyze_promotion(orders, promo_start, promo_end, baseline_days=30, promo_name="")` -> AnalyticsResult (R6.1, D-04, D-05):**

    - `promo_start` and `promo_end` are `datetime.date` objects.
    - Partition orders by `local_date` into: baseline (promo_start - baseline_days to promo_start - 1 day), promo (promo_start to promo_end inclusive), post-promo (promo_end + 1 to promo_end + promo_duration).
    - Compute daily averages for baseline and promo periods for: revenue, orders, units, AOV, discount_rate, units_per_order using `aggregate_metrics`.
    - Compute lift % = `(promo_daily - baseline_daily) / baseline_daily * 100` for each metric.
    - **Section 1 "Promotion Impact"**: table with columns [Metric, Baseline (daily avg), Promo (daily avg), Change, Lift %].
    - **Section 2 "Post-Promo Impact" (D-04)**: Compare post-promo daily avg to baseline daily avg for revenue and orders. Show dip/recovery. Columns [Metric, Baseline (daily avg), Post-Promo (daily avg), Change %].
    - **Section 3 "Product Cannibalization" (D-05)**: For each product_id, compute revenue share (% of total revenue) during promo vs baseline. Flag products where share dropped >2pp as "cannibalized". Table columns [Product, Baseline Share %, Promo Share %, Change pp].
    - Summary: "Revenue lifted X% during {promo_name or 'the promotion'} ({promo_start} to {promo_end}). Post-promo revenue dipped Y% vs baseline."
    - Recommendations (D-02): Generate 1-2 based on findings. E.g., if hangover is significant: "Consider a shorter promo to reduce post-promo dip." If cannibalization detected: "Products X, Y lost share during promo -- consider excluding from discount."

    **2. `detect_anomalies(series, forecast_result, sensitivity="medium", lookback_days=90, known_events=None)` -> AnalyticsResult (R6.2, D-06 through D-13):**

    - `series`: pd.Series (daily actuals with DatetimeIndex).
    - `forecast_result`: ForecastResult with confidence_bands containing quantile channels (from a backcast or forward forecast over the historical period).
    - `sensitivity`: "low" | "medium" | "high". Maps to bands per D-06: low=q10/q90, medium=q20/q80, high=q30/q70.
    - `lookback_days`: int, default 90. Auto-clamp to `min(lookback_days, len(series))` per D-13.
    - `known_events`: optional list of `{"date": "YYYY-MM-DD", "label": "Black Friday"}` dicts for labeling per D-09.
    - If `len(series) < 90`, prepend warning to summary per D-07: "Limited history ({N} days) -- anomaly detection may be less reliable."
    - Compare actuals against selected quantile bands. For each day outside bands, record: date, actual, expected (mean channel), lower bound, upper bound.
    - Cluster consecutive anomaly days per D-08: if gap between anomaly days <= 1, merge into single cluster.
    - For each cluster, compute: date range (start-end), summed actual, summed expected, avg deviation %, direction ("Spike" or "Drop") per D-10.
    - Label clusters per D-09: check if any date in cluster overlaps a known_event date.
    - Table per D-11: columns [Date Range, Direction, Actual, Expected, Band, Deviation %].
    - Direction format per D-10: "Spike (+42% above expected)" or "Drop (-31% below expected)".
    - Single metric per call per D-12.
    - Recommendations: e.g., "3 anomalies detected in last 90 days. The Nov 29-Dec 1 spike aligns with Black Friday -- consider planning inventory."

    **3. `compare_periods(orders, period_a_start, period_a_end, period_b_start, period_b_end, metrics=None)` -> AnalyticsResult (R6.3, D-03):**

    - All date params are `datetime.date`.
    - `metrics` defaults to `SUPPORTED_METRICS` (all 6 per D-03).
    - For each metric, compute totals for period A and period B using `aggregate_metrics`.
    - Change = B - A. Change % = ((B - A) / A) * 100. Handle A=0 gracefully.
    - Table columns: [Metric, Period A, Period B, Change, Change %].
    - Bold/highlight biggest movers per D-03: prepend "**" and append "**" to the metric name cell for the metric(s) with the largest absolute change %.
    - Summary: "Period B vs Period A: Revenue {up/down} {X}%, Orders {up/down} {Y}%."
    - Recommendations: e.g., "AOV increased 15% -- the product mix shift is worth investigating."

    **4. `get_seasonality(series, granularity="day_of_week")` -> AnalyticsResult (D-14):**

    - `series`: pd.Series with DatetimeIndex.
    - `granularity`: "day_of_week" | "monthly" | "quarterly".
    - Compute index: group by granularity bucket, compute mean per bucket, normalize so overall mean = 100. Index = (bucket_mean / overall_mean) * 100.
    - For day_of_week: rows are Monday..Sunday.
    - For monthly: rows are January..December. Warn if <12 months of data.
    - For quarterly: rows are Q1..Q4. Warn if <2 years of data.
    - Table per D-14: columns [Period, Index, Interpretation]. Interpretation: >110 = "Above average", 90-110 = "Average", <90 = "Below average".
    - Summary: identify peak and trough. E.g., "Saturday is the strongest day (index 134), Tuesday is weakest (index 78)."
    - Recommendations: e.g., "Consider running promotions on slower days (Tuesday, Wednesday) to balance demand."

    **5. `cohort_retention(orders, cohort_period="monthly", periods_out=6)` -> AnalyticsResult (R6.4, D-15):**

    - `orders`: list of normalized order dicts (must include `customer_id` and `local_date`).
    - Assign each customer to a cohort based on their first purchase date (month for "monthly", week for "weekly").
    - For each cohort, track how many customers purchased again in subsequent periods (period 1, 2, ..., periods_out).
    - Retention rate = (customers purchasing in period N) / (cohort size) * 100.
    - Compute avg LTV per cohort = total revenue from cohort customers / cohort size.
    - Table per D-15: rows = cohort labels (e.g., "2025-01", "2025-02"), columns = [Cohort, Size, P1%, P2%, ..., P{N}%, Avg LTV].
    - Summary row: average retention across all cohorts per period.
    - Summary: "Average 30-day retention is X%. Best cohort: {month} at Y%."
    - Recommendations: e.g., "Retention drops sharply after period 2 -- consider a re-engagement campaign at the 60-day mark."

| Boundary | Description |
|----------|-------------|
| User input -> analytics functions | Date ranges, sensitivity levels, promo names from tool params |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | analytics.py date params | mitigate | All date params are datetime.date objects (type-safe). Validate end >= start in each function, return error AnalyticsResult if invalid. |
| T-05-02 | DoS | compare_periods with huge date range | accept | Orders are already fetched and bounded by Shopify client's date range. Analytics functions operate on in-memory data only. |
| T-05-03 | Information Disclosure | customer_id in normalize output | accept | customer_id is a Shopify GID (opaque numeric ID), not PII. Only used for cohort grouping, never surfaced in tool output. |
| T-05-04 | Tampering | promo_name in analyze_promotion | mitigate | promo_name is only used in output labels via f-string in markdown. Never used in queries, file paths, or code execution. |
</threat_model>
