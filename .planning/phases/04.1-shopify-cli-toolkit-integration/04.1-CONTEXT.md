**Gathered:** 2026-04-16

<domain>
## Phase Boundary

Replace the httpx-based Shopify client internals with a dual-backend architecture: Shopify CLI (`shopify store execute`) as the primary backend for interactive use, direct httpx as fallback for headless/CI/Docker. Add browser-based OAuth via `shopify store auth`. The public API of `ShopifyClient` (fetch_orders, fetch_products, etc.) does NOT change — only the internal wiring.

</domain>

<decisions>
## Implementation Decisions

### Backend Protocol
- `ShopifyBackend` Protocol class with `post_graphql`, `post_graphql_mutation`, `download_url`, `close` methods
- `CliBackend` implementation: shells out to `shopify store execute --json --query`
- `DirectBackend` implementation: wraps existing httpx code (current behavior)
- `create_backend(settings)` factory: token in env → DirectBackend; CLI on PATH → CliBackend; neither → clear error

### Query Execution (CLI path)
- `execute_graphql()` in new `core/shopify_exec.py` module
- Uses `asyncio.create_subprocess_exec` to call `shopify store execute`
- `--json` flag for parseable output, `--allow-mutations` for bulk ops
- `--store` flag always included with the store domain
- Parse stdout as JSON, stderr as error message

### Config Changes
- `access_token` becomes `SecretStr | None = None` (was required)
- `shop` stays required (needed for both backends)
- All other settings unchanged

### Bulk Operations
- CLI backend handles the GraphQL mutation (start) and polling (poll) via `shopify store execute --allow-mutations`
- JSONL download stays as plain `httpx.get(url)` — the URL is a signed S3-style link, no Shopify auth needed
- `httpx` remains a dependency for this one use case

### Auth CLI Command
- New `auth` subcommand on `shopify-forecast` CLI
- Wraps `shopify store auth --store <store> --scopes read_orders,read_all_orders,read_products,read_inventory`
- Runs a quick `shop { ianaTimezone }` query to verify auth worked
- Subprocess call (not async — this is a one-shot user-facing command)

### Testing
- CLI backend: mock `asyncio.create_subprocess_exec` to simulate `shopify store execute` output
- Direct backend: keep existing `respx` mocks unchanged
- Factory: test backend selection logic based on Settings + shutil.which("shopify")
- Integration: `@pytest.mark.integration` test that runs a real `shopify store execute` if CLI is available

### Claude's Discretion
- Exact error message formatting for CLI-not-found and auth-failed scenarios
- Whether to use `shutil.which("shopify")` or subprocess check for CLI detection
- Internal module structure within `shopify_backend.py` (single file vs split)
- Whether `CliBackend` needs connection pooling or retry logic (CLI handles this)

</decisions>

<specifics>
## Specific Ideas

- The `shopify store execute` command returns JSON when `--json` flag is used
- Shopify CLI stores auth credentials internally after `shopify store auth` — no token management needed
- Subprocess overhead (~100-200ms per call) makes bulk ops mandatory for large stores (>1000 orders)
- The `--allow-mutations` flag is required for `bulkOperationRunQuery` mutation

</specifics>
