  <action>
Look up a current commit SHA from the google-research/timesfm master branch. Do NOT hardcode a SHA from research — verify it live now (research flagged this LOW confidence).

Preferred method (no auth, no clone):
```bash
git ls-remote https://github.com/google-research/timesfm.git HEAD
```
This prints `<sha>\tHEAD`. Take the 40-char SHA.

Fallback (if git ls-remote unavailable):
```bash
curl -s https://api.github.com/repos/google-research/timesfm/commits/master | python -c "import json,sys; print(json.load(sys.stdin)['sha'])"
```

Sanity checks on the chosen SHA:
1. Confirm the commit exists: `curl -sI https://github.com/google-research/timesfm/commit/<SHA>` returns `200`.
2. Confirm the `TimesFM_2p5_200M_torch` symbol exists at that SHA:
   ```bash
   curl -s "https://raw.githubusercontent.com/google-research/timesfm/<SHA>/src/timesfm/__init__.py" | grep -q "TimesFM_2p5_200M_torch"
   ```
   If this grep fails, the symbol moved — pick a different recent master SHA and retry, or search the tree: `curl -s "https://api.github.com/repos/google-research/timesfm/git/trees/<SHA>?recursive=1" | grep timesfm_2p5_torch`.

Write the full 40-char SHA to `.planning/phases/01-scaffold-config/timesfm-sha.txt` on a single line (no trailing newline issues, no quotes). Task 2 will read it back.

If no valid SHA can be located (network failure, repo restructure), STOP and escalate — do not invent a SHA. Phase 3 depends on this being correct.

Note: no `[torch,xreg]` extras on the timesfm dep — research/timesfm.md confirms XReg is Phase 5, not Phase 1. Keep the dep as a bare git URL; extras can be added later if needed.

2. Dev optional-dependencies — add after `[project.scripts]`:
```toml

gpu = []
dev = [
  "pytest>=8",
  "pytest-asyncio>=0.24",
  "respx>=0.22",
  "ruff>=0.6",
  "mypy>=1.11",
]
```

3. uv configuration — add:
```toml
[tool.uv]
default-groups = ["dev"]

[[tool.uv.index]]
name = "pytorch-cpu"
url = "https://download.pytorch.org/whl/cpu"
explicit = true

[tool.uv.sources]
torch = [
  { index = "pytorch-cpu", marker = "sys_platform == 'linux' or sys_platform == 'win32'" },
]
```

Do NOT use `[tool.uv.sources]` for `timesfm` — a git dep in `dependencies` is already resolved directly.

4. Pytest + ruff + mypy tool tables (from research/packaging.md — add as a single block):
```toml
[tool.pytest.ini_options]
minversion = "8.0"
asyncio_mode = "strict"
testpaths = ["tests"]
addopts = "-ra --strict-markers --strict-config"
pythonpath = ["src"]

[tool.ruff]
line-length = 100
target-version = "py311"

[tool.mypy]
python_version = "3.11"
strict = true
```

5. Run `uv sync` to resolve and lock. This will:
   - Generate `uv.lock`
   - Download torch from the CPU index on Linux (on macOS, falls through to PyPI — expected)
   - Clone timesfm at the pinned SHA into the venv

   If `uv sync` fails on the timesfm dep (e.g., build error from a transitive), try `uv sync --no-install-project` to isolate — but a hard failure means the SHA picked a bad state and Task 1 must be rerun with a different SHA.

   If `uv sync` fails on torch wheel resolution on macOS, confirm the marker syntax excludes darwin — macOS must resolve through PyPI (`sys_platform == 'darwin'` is not matched by the override, which is correct).

6. Confirm the entry points are installable by running `uv run shopify-forecast` and `uv run shopify-forecast-mcp` — both should print "not implemented" to stderr and exit 0.
